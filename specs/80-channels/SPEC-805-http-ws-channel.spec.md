# HTTP loopback RPC channel with bearer auth

## 1. Outcomes

- nimbus daemon subcommands (`nimbus cost`, `nimbus health`) communicate with the daemon over loopback HTTP RPC.
- Binds exclusively to `127.0.0.1`; remote bind is refused at startup.
- Bearer token auth (`nmbt_<base64url>`) per workspace, stored in SPEC-152 vault.
- Minimal footprint: no WebSocket, no pairing flow, no IP ban tracker.

## 2. Scope

### 2.1 In-scope
- `Bun.serve` HTTP server, loopback-only (`127.0.0.1`; remote bind rejected at startup)
- `POST /api/v1/messages` — send a message, receive acknowledgement JSON
- `GET /api/v1/health` — health check (no auth required)
- Bearer token format `nmbt_<base64url(32 bytes)>` generated by `auth.ts`; stored in vault under `http.bearerToken.<workspaceId>`
- `auth.ts`: `generateBearerToken`, `storeBearerToken`, `verifyBearer` (constant-time compare)
- Token masked as `nmbt_***<last4>` in all log output

### 2.2 Out-of-scope (cut at v0.3.1 review — 1-user local-first, no remote-client story yet)
- **WebSocket streaming** (`wsStream.ts` deleted) — out-of-scope until v0.4+ remote-client user story validates demand
- **Device pairing flow** (`pairing.ts` deleted) — 6-digit code + QR + PBKDF2 session store; out-of-scope until v0.4+
- **IP ban / LRU rate limiter** (`isIpBanned`, `recordFailedAuth`, ip map) — loopback-only means attacker is already local; out-of-scope until v0.4+
- **TLS config** (`tlsCert`, `tlsKey`) — loopback does not need TLS; out-of-scope until remote bind is unlocked
- **`trustedProxy` / X-Forwarded-For extraction** — meaningless on loopback; out-of-scope
- mTLS / client certificates → v0.4
- OAuth2 device flow → v0.5
- HTTP/2 server push → not planned

## 3. Constraints

### Technical
- Bun ≥1.2 `Bun.serve` (HTTP only; no `ws` npm package)
- No `any` types; strict TypeScript
- Max 400 LoC per file; `server.ts` ≤120 LoC; `auth.ts` ≤60 LoC
- Bearer token comparison uses timing-safe byte comparison to prevent timing attacks
- Remote bind → `NimbusError(ErrorCode.X_NETWORK_BLOCKED, {reason: 'remote_bind_not_allowed'})`

### Security
- Bearer token in `Authorization: Bearer <token>` header only
- Token never logged in plaintext; masked as `nmbt_***<last4>`
- Loopback-only bind: attacker must already be local (same security boundary as vault)

### Performance
- Auth check (vault lookup + compare) <5ms warm

### Resource / Business
- Port configurable via `--port` (default 7432)

## 4. Prior Decisions

- **`Bun.serve` not `express`/`fastify`** — Bun-native HTTP; eliminates extra deps; aligns with project Bun-native rule
- **`nmbt_` token prefix** — prefix makes tokens greppable in logs for accidental exposure; matches project `SENSITIVE_FIELDS` pattern `*_token`
- **Loopback-only in v0.3** — nimbus is 1-user local-first; no remote client use case exists yet; TLS + IP-ban complexity is net-negative until v0.4 validates demand
- **v0.3.1 scope-down (2026-04-16)** — after critical review, cut WS streaming (`wsStream.ts`), device pairing (`pairing.ts`), IP ban LRU, TLS gate, and `trustedProxy` from v0.3 scope. ~580 LoC removed. Features preserved as out-of-scope (§2.2) for v0.4+ if remote-client user story validates. Net result: ~120 LoC loopback RPC only.

## 6. Verification

### 6.1 Unit Tests
- `server.test.ts`:
  - `GET /api/v1/health` → 200 without auth
  - `POST /api/v1/messages` with valid bearer → 200 + EventBus inbound published
  - `POST /api/v1/messages` with invalid bearer → 401
  - `POST /api/v1/messages` missing `text` field → 400
  - Unknown route → 404
  - Remote bind (`0.0.0.0`) → startup throws `NimbusError`
  - `stop()` → subsequent requests connection-refused
- `auth.test.ts`:
  - `generateBearerToken` → `nmbt_<base64url>` pattern; 20 tokens unique
  - `verifyBearer` → true for valid; false for wrong; false for unstored
  - Timing: valid vs invalid average difference <5ms
  - `maskToken` → shows prefix + last 4 only

### 6.2 E2E Tests
- `tests/e2e/http-channel.test.ts`: start server on random port; POST message → verify response

### 6.3 Performance Budgets
- Auth check (vault lookup + compare) <5ms warm

### 6.4 Security Checks
- Bearer token stored in vault (SPEC-152); never in workspace config file
- Token logged only as `nmbt_***<last4>` in all log output
- Bind address checked at `createHttpChannel` call time; non-loopback → hard error before `Bun.serve` called

## 7. Interfaces

```ts
// server.ts
export interface HttpChannelConfig {
  port?: number        // default 7432
  bindAddress?: string // default '127.0.0.1'; must be loopback
}

export interface HttpChannelAdapter extends ChannelAdapter {
  readonly boundPort: number
}

export function createHttpChannel(
  cfg: HttpChannelConfig,
  channelManager: ChannelManager,
  workspaceId: string,
): HttpChannelAdapter

// auth.ts
export function generateBearerToken(): string                                    // returns nmbt_<base64url>
export function storeBearerToken(workspaceId: string, token: string): Promise<void>
export function verifyBearer(token: string, workspaceId: string): Promise<boolean>
export function maskToken(token: string): string                                 // nmbt_***<last4>
```
