# HTTP/WS channel with bearer auth and device pairing

## 1. Outcomes

- Remote clients (CLI, mobile, custom tooling) communicate with nimbus over HTTP POST and WebSocket streaming.
- Default bind `127.0.0.1`; remote bind requires explicit `--bind 0.0.0.0` flag AND TLS config.
- Bearer token auth (`nmbt_<base64url>`) per workspace, stored in SPEC-152 vault.
- Device pairing flow (6-digit code + QR, 5 min TTL) enables first-time token exchange without manual copy-paste.
- Brute-force protection: 10 failed auth attempts per IP per minute → 15 min ban.

## 2. Scope

### 2.1 In-scope
- `Bun.serve` HTTP + WebSocket server (single port, protocol upgrade on `/api/v1/stream`)
- `POST /api/v1/messages` — send a message, receive full response as JSON
- `WS /api/v1/stream` — send a message, receive token-streaming response as NDJSON frames
- Bearer token format `nmbt_<base64url(32 bytes)>` generated by `auth.ts`; stored in vault under `http.bearerToken.<workspaceId>`
- `pairing.ts`: 6-digit numeric code (CSPRNG), QR rendered via `qrcode` package to stdout, 5 min TTL; one-time exchange returns bearer token
- IP-based rate limiter for failed auth: 10 failures/min per IP → `BanEntry` with 15 min expiry (in-memory, cleared on restart)
- `auth.ts`: `verifyBearer(token, workspaceId)` constant-time compare; logs failed attempts to SPEC-601 security log
- TLS: when `--tls-cert` + `--tls-key` flags provided, `Bun.serve` uses `tls` option; remote bind without TLS → startup error

### 2.2 Out-of-scope
- mTLS / client certificates → v0.4
- OAuth2 device flow → v0.5
- Rate limiting per-user (not just per-IP) → v0.4
- HTTP/2 server push → not planned

## 3. Constraints

### Technical
- Bun ≥1.2 `Bun.serve` (native WS; no `ws` npm package)
- No `any` types; strict TypeScript
- Max 400 LoC per file; `server.ts` ≤250 LoC; overflow into `wsStream.ts`
- Bearer token comparison uses `crypto.timingSafeEqual` to prevent timing attacks
- Remote bind without TLS → `NimbusError(ErrorCode.P_PERMISSION_DENIED, {reason: 'remote bind requires TLS'})`

### Performance
- First WS token frame delivered <200ms after message receipt
- Auth check <5ms per request

### Resource / Business
- Port configurable via `--port` (default 7432)
- QR display requires terminal with Unicode support (skip QR gracefully if `NO_COLOR=1` or non-TTY)

## 4. Prior Decisions

- **`Bun.serve` not `express`/`fastify`** — Bun-native WS + HTTP in one server; eliminates extra deps; aligns with project Bun-native rule
- **`nmbt_` token prefix** — prefix makes tokens greppable in logs for accidental exposure; matches project `SENSITIVE_FIELDS` pattern `*_token`
- **6-digit code not URL-based pairing** — 6-digit code works in any terminal/SSH session without a browser; URL-based pairing requires browser, conflicting with headless use case
- **In-memory IP ban not Redis** — nimbus-os is single-process local; Redis dependency contradicts local-first principle; on restart ban list resets (acceptable for v0.3)
- **Single port for HTTP + WS** — simplifies firewall rules and user config; Bun upgrade path is clean

## 6. Verification

### 6.1 Unit Tests
- `server.test.ts`:
  - Valid bearer token → 200; invalid token → 401; 11th invalid from same IP → 429
  - `POST /api/v1/messages` with valid body → `channel.inbound` published to EventBus
  - Remote bind without TLS config → startup throws `NimbusError`
  - `stop()` → all WS connections closed gracefully
- `auth.test.ts`: `verifyBearer` returns false for wrong token; timing: 1000 iterations vary <10µs (no early exit)
- `pairing.test.ts`: code length 6; expired after 5 min (mock `Date.now`); reuse rejected

### 6.2 E2E Tests
- `tests/e2e/http-channel.test.ts`: start server on random port; POST message → verify response; open WS → stream 3 tokens → close

### 6.3 Performance Budgets
- Auth check (vault lookup + compare) <5ms warm
- First WS frame <200ms after `channel.inbound` published

### 6.4 Security Checks
- Bearer token stored in vault (SPEC-152); never in workspace config file
- Token logged only as `nmbt_***…` (last 4 chars shown) in audit log
- IP extracted from `X-Forwarded-For` only when `--trusted-proxy` flag set; default: use direct connection IP
- TLS cert/key paths validated for existence and `0600` mode before `Bun.serve` starts
- Pairing code stored as `PBKDF2(code, salt, 10000, 32)` in memory (not plaintext) to prevent leakage via core dump

## 7. Interfaces

```ts
export interface HttpChannelConfig {
  port?: number               // default 7432
  bindAddress?: string        // default '127.0.0.1'
  tlsCert?: string            // path to PEM cert (required for non-loopback)
  tlsKey?: string             // path to PEM key
  trustedProxy?: boolean      // enable X-Forwarded-For extraction
}

export function createHttpChannel(cfg: HttpChannelConfig): ChannelAdapter

// auth.ts
export function generateBearerToken(): string           // returns nmbt_<base64url>
export function verifyBearer(token: string, workspaceId: string): Promise<boolean>
export function recordFailedAuth(ip: string): { banned: boolean; remainingAttempts: number }

// pairing.ts
export interface PairingSession {
  code: string
  expiresAt: number
  workspaceId: string
}
export function createPairingSession(workspaceId: string): PairingSession
export function redeemPairingCode(code: string): Promise<string | null>   // returns bearerToken or null
```
